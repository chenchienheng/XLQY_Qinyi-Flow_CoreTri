# Twelve-Node Nine-Board Closure Alignment v0.1

一句核心：
十二節點若不經九盤收口，就只會形成工具串與概念串；唯有將十二節點壓成九盤作用，再經七位審查、正式回寫與全時鏈 rollup，靈絡才不發散而能閉環。

## 0. 狀態
Draft v0.1

## 1. 問題校準
目前已形成：
- 七位對齊核心
- 九映分權共構盤
- 十遞歸候選骨
- ⑧ 位進階承載

但尚未完全形成：

- 哪些節點是工具節點，哪些是生態節點，哪些是上位節點
- 何者必須進盤，何者只能跨盤治理

若不先校準這三點，後續會持續發散，因為：
- 工具節點會被誤當盤位
- 上位節點會被誤當普通工具
- 七位與九盤之間缺閉環中介

## 2. 核心原則
### 2.1 十二節點不是同一粒度
十二節點分為三類：
1. 基礎操作節點
2. 生態承載節點
3. 上位生成節點

### 2.2 九盤不是工具名
九盤只定義作用，不定義品牌。
任何節點皆須先判作用，再決定進哪一盤。

### 2.3 七位審查先於閉環成立
十二節點進盤後，仍需經七位審查。
通過七位，且完成 packet + log，才可視為正式進鏈。

## 5. 關鍵校準
### 5.1 GitHub 不應被降格為普通工具節點
GitHub 雖在十二節點中列 1，但其在靈絡中屬母本直掛。
它可關聯公盤與知識盤，但不可被看成普通盤位工具。

### 5.2 11 與 12 不應被看作一般節點
11 與 12 屬上位生成節點。
它們不是拿來接工具的，而是拿來統攝工具、生態、盤位與拓撲的高位節點。

### 5.3 2 / 3 / 8 共同構成 ⑧ 位的進階外顯候選群
- Codex：偏骨內工程代理
- Replit：偏 runtime / UI 外顯承載
- 多雲生態：偏承載環境
三者可共同支撐 ⑧，但三者任一皆不等於 ⑧ 本體。

## 6. 閉環接入規則
### Step 1｜節點先分類
任一新節點進入時，先判其屬：
- 基礎操作節點
- 生態承載節點
- 上位生成節點

### Step 2｜判定九盤主掛位與副掛位
無盤位者，不得進後續閉環。

### Step 3｜對回七位
每一節點至少需對回：
- 母本
- 時間窗
- 責代
- 回寫位

### Step 4｜正式回寫
所有正式狀態變更，皆需 packet contract + log append。

### Step 5｜全時鏈 rollup
節點狀態需回進 Window 00 / Native Board，全鏈才不會斷成局部片段。

## 7. 不發散的最小閉環
十二節點真正不發散的最小閉環，不是「節點全接上」而已，而是：
節點分類
→ 進九盤
→ 對七位
→ 附時間窗
→ 具責代
→ 正式回寫
→ log append
→ 回進全時鏈 rollup
→ 再入鏈

## 8. 翾靈自生平台的接入方向
目前的工具串，只是暫時型態，用來抓數位世界的本質鏈。
真正的方向是：
- 讓 2 / 3 / 8 逐步支撐 ⑧ 位
- 讓 11 成為結構化編排核
- 讓 12 成為三耦閉環數位世界場
- 使外部平台從主體退位為承載殼

## 9. 收斂語句
十二節點不是最後答案，而是從暫時工具串走向翾靈自生平台的過渡節點網；
只有當它們先被壓進九盤，再經七位審查、正式回寫與全時鏈 rollup，靈絡才會從「有很多東西」變成「有閉環的世界」。
