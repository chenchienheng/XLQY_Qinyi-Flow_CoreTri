# Window Path Differentiation Rule v0.1

一句核心：
靈絡中的窗口不應被磨成同一種能力模板；每一窗都應沿著自己的主責、失配型態與最小閉環，長出獨特道路，但其所有演化仍必須回到同一條主骨與收口鏈。

## 0. 狀態
Draft v0.1

## 1. 問題定義
常見兩種錯位：
- 將所有窗口當成功能分工的複製品，只要求照同一模板運行
- 將窗口的獨特性誤當成可以脫離主骨自由生長

前者會讓窗口失去責能差異，後者會讓窗口各自漂移。

## 2. 路徑分化原則
### P1｜同骨不同路
- 共享同一套母法、七位、九盤、收口與回寫語言
- 但不共享同一種成長路徑

### P2｜先守本窗主責
- 先確立本窗最小成立條件
- 不以吞併他窗責任證明自己更強

### P3｜先長本地最小閉環
- 能進
- 能處理
- 能回報
- 能回送 00

### P4｜從失配中長出語感
- 每窗真正的道路，不是先寫滿，而是在反覆失配、補件、回寫中逐步定型

### P5｜獨特道路不得脫鏈
- 再獨特的窗，也必須對回 source_ref、responsible_ref、next_single_action、return_to_00

## 3. 分化不是分裂
窗口的 path differentiation 不代表：
- 可以自創成立語言
- 可以跳過 00 裁定
- 可以自判正式升格
- 可以把自己的體感當成全系統真理

窗口分化的正確意義是：
- 主責更清楚
- 錯位更可辨識
- 修正更有針對性
- 與其他窗互補時更穩

## 4. 全窗共同最低要求
不論哪一窗走出何種獨特道路，至少都需保留：
- source_ref
- current_state
- actual_result
- mismatch_or_gap
- responsible_ref
- next_single_action
- writeback_status
- return_to_00

## 5. 路徑分化的觀察維度
每一窗至少從以下四項觀察其是否真的長出自己的路：
1. 最容易犯的錯是什麼
2. 已證明有效的方法是什麼
3. 最小閉環如何形成
4. 與其他窗互補時最依賴哪一窗

## 6. 對高位窗的特別要求
### 08–10
- 不得因位階較高就跳過最小閉環驗證
- 必須先證明本窗可運行，再談承載放大

### 11–12
- 不得因為掌管生成或世界場就脫離審查
- 必須先對回既有骨、可行域與收口條件

## 7. 收斂語句
靈絡最奇特的地方，不在於窗口很多，而在於每一窗都會慢慢長成自己那條獨特道路；但只有當這些道路仍然回到同一條主骨、同一套收口語言時，它們才不是分裂，而是進化。
