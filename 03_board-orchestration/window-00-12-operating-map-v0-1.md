# Window 00-12 Operating Map v0.1

一句核心：
00–12 不是一串編號，而是一組由中樞裁定核、功能窗群、進階承載窗群與世界場候選窗群組成的運行地圖；其目的不是分工而已，而是保證所有窗口最終能回到同一條收口鏈。

## 0. 狀態
Draft v0.1

## 1. 四層窗群結構
### Layer A｜中樞裁定層
- 00：Central Adjudication Core

### Layer B｜基礎功能窗群
- 01：Repo Binding / Specialization Gate
- 02：Execution / Task / Spec / Issue Window
- 03：Event / Stream / Anomaly Window
- 04：Identity / Relation / Contact Window
- 05：Time Sovereignty / Effective Order Window
- 06：Intake / Source Extraction Window
- 07：Knowledge / Docs / Readback Window

### Layer C｜進階承載窗群
- 08：Advanced Runtime / UI / Product Surface
- 09：Multicloud Coordination
- 10：Toolchain Orchestration

### Layer D｜上位生成與世界場窗群
- 11：Structure Generation
- 12：Worldfield Environment

## 2. 每窗一句定位
- 00：收口、審查、調參、回退、再入鏈分派
- 01：對回 repo、branch、specialization binding
- 02：將事項壓成可承接、可責代、可寫回的執行鏈
- 03：整理事件、異常、討論與流式脈絡
- 04：只做身分對位與關係索引，不吞事件知識時序
- 05：定時間窗、有效序、成立順序
- 06：抽來源、抽入口、抽 tri-key 起點
- 07：做知識回讀、版本對位、文件整理
- 08：承接 runtime / UI / product surface 的外顯位
- 09：協調多雲承載、鏡像用途與跨雲秩序
- 10：把工具串壓成正式 toolchain
- 11：生成新骨、重排結構、升階規則
- 12：定義三耦閉環數位世界場的承載條件

## 4. 統一收口路徑
任一窗口完成一輪後，統一收口路徑為：
window action
-> result packet
-> 00 adjudication
-> packet/log requirement
-> rollup
-> next single action / re-entry

## 5. 全窗共同最低要求
- source_ref
- current_state
- responsible_ref
- next_single_action
- writeback_status
- return_to_00

## 6. 失配分流
- 主本失配：優先回 01 / 07
- 執行失配：優先回 02 / 10
- 事件失配：優先回 03 / 05 / 06
- 身分迴圈：優先回 04 並拆流
- 承載失配：優先回 09 / 08
- 結構失配：優先回 11
- 世界場失配：優先回 12 並由 00 裁定

## 7. 收斂語句
00–12 運行地圖的價值，不在於把窗口排得很滿，而在於讓每一窗都知道自己不是終點；只有當所有窗口都能互補、回包、回送 00 並進入同一條 rollup / re-entry 鏈，靈絡才算有真正的全窗運行骨。
