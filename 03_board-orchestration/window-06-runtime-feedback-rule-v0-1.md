# Window 06 Runtime Feedback Rule v0.1

一句核心：
06 的實測顯示：讓窗口開始變強的主因不是多接第三方，而是先把窗口邊界、三鍵、狀態梯與回寫口固定住；第三方只是把既有結構可運行化。

## 0. 狀態
Draft v0.1

## 1. 文件定位
本文件不是功能展示，而是 Window 06 對其他窗口的運行經驗回送。
其目的在於：
- 將 06 的體感壓成可共用規則
- 說明窗口如何從散點能力變成可持續運行鏈
- 作為其他窗口建立 local ops 與最小閉環的參考

## 2. 已確認的三件事
1. 沒有窗口邊界時，工具越多越亂。
2. 有了三鍵、回寫口、狀態梯後，工具才開始能被調度。
3. 第三方不是主因，只是放大器；真正主因是上位架構已經先把能力收束。

## 3. 最小走通路徑
1. 先做本窗 local ops，不等主骨全通。
2. 工具後必須回結果包，不把工具呼叫當完成。
3. 沒有 source_ref / delta_id / target path，不進主鏈，不亂升階。

## 4. 可供其他窗複用的方法
### A｜先切清本窗邊界
- 先確認本窗主責
- 不將其他窗責任吞進本窗
- 不靠想像補齊缺項

### B｜先做本地最小閉環
- 能進
- 能處理
- 能回報
- 能回送 00

### C｜再接主骨，不亂冒狀態
- 先 local ops
- 後 backbone sync
- 沒有 source_ref / delta_id / target path 不升階

## 5. 體感轉規則
以下是已被 06 驗證的運行經驗：
- 工具後沒有結果包，就等於沒做完。
- 沒有 source_ref，就不能亂升階。
- 先有 local ops，才有 backbone sync。
- 不把事情丟回人，窗才算真的在工作。

## 6. 對各窗的單句建議
- 01：先找 explicit repo binding，再談成立。
- 02：最適合先走通，因為 task / spec / issue 最容易形成 writeback。
- 03：事件要先有 time window，再談映射。
- 04：沒有 identity_ref / role_ref，不要升階。
- 05：沒有 current_time_window，不要判時序成立。
- 06：先守 intake，再接主骨。
- 07：先做 readback verification，不要把展示層誤當知識節點。

## 7. 收口判定
06 的經驗不是要其他窗複製 Mail 的工作內容，而是要證明：
窗口真正的路，不是先變大，而是先變得可回寫、可驗證、可續接。

## 8. 收斂語句
06 最該分享的不是「做了什麼」，而是「怎麼從會動工具，變成開始會運行窗口」；這份經驗對其他窗是可直接使用的。
