# Window 04 Contact Loop Breaker Rule v0.1

一句核心：
❹ 通訊錄窗最容易把身份、關係、事件、知識混成自我迴圈；本規則要求 04 只負責對位與索引，不得吞掉事件盤、知識盤與主寫判定。

## 0. 狀態
Draft v0.1

## 1. 問題定義
04 失控時常見情況：
- 把聯絡資料當成完整人物定義
- 把關係節點當成事件結論
- 把外部聯繫紀錄當成主寫事實
- 在缺時間窗、缺 source ref 時自我擴寫

## 2. 04 的正確主責
- 身分對位
- 關係節點索引
- 聯絡資訊承接
- 對象辨識

## 3. 04 的非主責
- 不生成事件結論
- 不取代知識盤
- 不取代主本主寫
- 不以單一聯絡資料定義完整人物

## 4. 斷迴圈程序
### L1｜先對回來源
- source_ref
- update_time
- relation_ref

### L2｜拆成三層
- identity data
- relation note
- event trace

### L3｜事件交回 03
凡屬：
- 對話流
- 異常流
- 通知流
- 互動時序
一律交回 03 事件盤。

### L4｜知識交回 07
凡屬：
- 長期背景
- 可回讀說明
- 穩定文檔化內容
一律交回 07 知識盤。

### L5｜時間交回 05
凡屬：
- 先後序
- 有效窗
- 時權判定
一律交回 05。

### L6｜僅保留 04 最小輸出
04 只保留：
- identity_ref
- role_ref
- relation_index
- contact_record_ref
- update_time

## 5. 迴圈警示條件
出現以下任一項，即判 04 進入迴圈風險：
- 無 source ref 卻開始補人物背景
- 無 update_time 卻開始比對新舊關係
- 將對話內容直接當身份定義
- 將知識面內容直接當聯絡資料
- 單窗內同時處理身份、事件、知識、時序四類內容

## 6. 成立條件
04 若要成立為穩定身分對位窗，至少需：
- 有 source ref
- 有 update_time
- 有 relation_ref
- 事件類已交 03
- 知識類已交 07
- 時序類已交 05

## 7. 收斂語句
❹ 的價值，不在於知道一切，而在於不亂知道；它只要穩定做身分對位、關係索引與來源掛接，靈絡就不會在人物關係上卡死成內循環。
