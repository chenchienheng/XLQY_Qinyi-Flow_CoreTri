# Contamination and Priority Policy

> Persistent note for handling earlier materials inside the evolving XLEN / Xuanling repository weave.
> This document does not assume that all earlier material is defective.
> It distinguishes between:
>
> - usable historical material
> - normal construction residue
> - structural contamination that distorts continuity
>
> It also defines a practical priority order for remediation.

## 1. Core Reading

Earlier materials should **not** be divided only into “useful” vs “useless”.
That reading is too crude.

A better reading is:

- some earlier materials are durable seeds
- some are normal construction-stage residue
- some contain structural contamination

Therefore the key question is not:
- should old material be discarded?

The key questions are:
- what kind of contamination is present?
- does it distort continuity, role, sovereignty, routing, or writeback?
- what priority should be assigned to correction?

---

## 2. What Counts as Contamination

Contamination here does **not** primarily mean emotional impurity or stylistic roughness.
It means material that introduces structural distortion into the weave.

### 2.1 Role Drift Contamination
Examples:
- treating stage notes as final sovereignty definitions
- reducing GitHub to bridge-only wording
- confusing common chain / public plate / mother architecture / subsystem

Effect:
- role confusion
- later sessions inherit wrong orientation

### 2.2 Layer Collapse Contamination
Examples:
- upper-order and stage-order mixed together
- interaction surface treated as sovereignty holder
- carrying mesh treated as structural bone

Effect:
- hierarchy collapse
- inability to sustain stable runtime reading

### 2.3 Writeback Loss Contamination
Examples:
- insights remain only in chat
- durable artifacts missing
- return path omitted

Effect:
- session reset
- repeated rediscovery cost

### 2.4 Flat Complexity Contamination
Examples:
- describing the architecture mainly as “complex”
- counting modules or files instead of cluster coverage and continuity quality

Effect:
- architecture appears inflated but not correctly read

### 2.5 Misbinding Contamination
Examples:
- unrelated materials woven together without role distinction
- subsystem-specific logic forced into common layer prematurely
- public plate confused with universal sameness

Effect:
- local specialization is damaged
- family structure becomes blurry

### 2.6 Unverified Assumption Contamination
Examples:
- claiming repository-wide completeness without full corpus verification
- treating suspected material as confirmed material

Effect:
- false certainty
- weak auditability

---

## 3. What Does **Not** Automatically Count as Contamination

The following do **not** automatically count as contamination:

- incomplete drafts
- stage notes
- exploratory vocabulary
- partial runtime descriptions
- unfinished system bindings

These may simply be:
- normal construction-state residue
- early seeds
- local subsystem fragments

They become contamination only when they are misread or left unclassified in ways that distort the broader system.

---

## 4. Priority Handling Logic

Priority should be assigned by structural damage potential, not by loudness or surface inconvenience.

### P0 — Sovereignty / Continuity Risk
Fix first when material threatens:
- time-chain reading
- return path
- writeback continuity
- repository re-anchor capability

Typical examples:
- no durable re-anchor note
- no runtime continuity map
- no returnable writeback artifact

### P1 — Structural Bone Risk
Fix next when material threatens:
- role orientation
- chain orientation
- structural bone reading
- durable classification

Typical examples:
- GitHub misread as flat storage only
- no chain master map
- no role classification table

### P2 — Routing / Weaving Risk
Fix when material threatens:
- corpus indexing
- subsystem/common-layer distinction
- external-node binding clarity
- discontinuity traceability

Typical examples:
- no corpus index
- no discontinuity register
- external node roles still mixed

### P3 — Expression / Optimization Risk
Fix later when material threatens:
- clarity
- naming precision
- local wording polish
- surface readability

Typical examples:
- wording redundancy
- local naming drift without structural damage
- incomplete explanatory polish

---

## 5. Handling Modes

Earlier material should not be handled with one action only.
Possible handling modes are:

### 5.1 Preserve
Use when material is structurally sound or still acts as a seed.

### 5.2 Reclassify
Use when the material is good but in the wrong role.

### 5.3 Isolate
Use when the material risks contaminating higher-order reading but still has historical or local value.

### 5.4 Rewrite / Rebind
Use when a core idea is valid but its current expression damages continuity.

### 5.5 Retire
Use only when material is both structurally misleading and no longer meaningful as seed, stage note, or record.

---

## 6. Practical Reading for Earlier Corpus

At the current stage, the safest reading is:

- assume earlier corpus contains a mix of seeds, residue, and contamination
- do not declare blanket purity
- do not declare blanket discard
- classify first, then decide action

Therefore the default rule is:

> classify before delete
> isolate before flatten
> rebind before abandon
