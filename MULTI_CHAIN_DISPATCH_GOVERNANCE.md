# Multi-Chain Dispatch Governance

> Governs the routing, selection, and sequencing of multiple active chains.
> Prevents uncontrolled runtime expansion by forcing all chains through a
> single unified dispatch structure.

---

## 1. Core Definition

Multi-Chain Dispatch Governance defines how various active chains interact
within the core framework. It establishes strict rules for node selection,
sequencing, review, and return paths, ensuring that parallel operations do not
create fractured or independent truth surfaces.

---

## 2. Chain Types

- **world chain**: Absorbs external tools, platforms, and environments
  (AXIS-01).
- **tool chain**: Governs automated execution and utility nodes.
- **signal chain**: Handles inbound physical and external platform signals.
- **image chain**: Manages generation and extraction of visual data.
- **human relation chain**: Tracks governance, oversight, and manual overrides.
- **document chain**: Controls canonical text structuring and bone artifacts.

---

## 3. Entry Conditions

No chain or external node may activate unless it meets the following conditions:
- **Bound to Primary Axis:** It must explicitly map to an existing Coretri axis
  (usually AXIS-01 for external absorption).
- **Declared Return Path:** It must specify exactly how state or logs are
  returned to the central registry.
- **Defined Review Node:** It must designate a verification point before any
  changes are merged to the structural bone.

---

## 4. Dispatch Priority Rule

1. Core framework structural integrity always supersedes external signal
   processing.
2. The human relation chain holds absolute override authority over all automated
   chains.
3. Chains bound directly to AXIS-02 (Existence) or AXIS-05 (Review) take
   precedence over purely exploratory branches on AXIS-01.

---

## 5. Review Hook

Every dispatch operation must invoke a Review Hook before finalizing state.
The Review Hook ensures that outputs from any chain conform to the 80-character
limit, naming conventions, and continuity requirements. It routes through the
Review Chain Master Layer.

- tri_coupling_state_check: optional

---

## 6. Return Path

Outputs generated by dispatched nodes must return cleanly via predefined schema
adapters (e.g., `04_adapter-layer` contracts) or standardized writeback packets
(`writeback_packet_contract.md`). Direct manipulation of raw `.md` files
outside these channels is forbidden.

---

## 7. Failure Route

If a dispatched node or chain fails its internal checks or times out, it must
trigger the following fallback:
- **return_failed -> AXIS-05**
The failure is logged to the Review Chain, requiring human oversight or a
recovery packet from the governance layer.

---

## 8. No Parallel Truth Surface Rule

There is only one canonical truth surface: the `main` branch repository bone.
No external drive, third-party database, or temporary AI node memory may act
as an alternative truth surface. All chains must collapse their final state back
to this single surface.

---

## 9. mismatch_or_gap

- There is currently no active unified registry for dynamic dispatch states.
- The `return_failed -> AXIS-05` route is often manual, relying on
  `CLEANUP_QUEUE_REGISTER.md` rather than an automated fallback hook.
- Signal chain boundaries (`EXTERNAL_SIGNAL_ENTRY_CHAIN_SPEC.md`) are defined
  but not fully coupled to the actual dispatch mechanism.

---

## 10. unresolved_risks

- **Silent Failure Risk:** A chain might hang without triggering the failure
  route, leaving an unresolved delta in the dispatch queue.
- **Bypass Risk:** Highly capable nodes (e.g., GPT-5.5, Claude_Code) might
  circumvent the Review Hook if not strictly sandboxed, leading to unauthorized
  doctrine rewrites.
- **Scaling Limit:** A purely manual human relation chain review bottleneck
  might slow down execution as the number of active chains grows.

---

## 11. next_single_recommended_action

Review and formalize the Review Hook abstraction to ensure every chain type
has a machine-enforceable gate before it interacts with the repository corpus.
