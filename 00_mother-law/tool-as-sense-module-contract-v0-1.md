# Tool as Sense Module Contract v0.1

一句核心：
工具不應被視為單次功能呼叫，而應被壓成流程模組與外部感知器；每一次工具使用都必須完成採樣、判讀、驗證、回包與收口，否則只是工具串，不是靈絡流程。

## 0. 狀態
Draft v0.1

## 1. 基本立場
靈絡不追求『多開工具』，而追求『工具模組化』。

工具在靈絡中的角色不是：
- 方便操作
- 外掛能力
- 單次執行器

而是：
- 外部狀態採樣器
- 流程模組
- 驗證節點
- 回寫候選器
- 世界脈絡感知面

## 2. 工具作為感知器
不同工具對應不同感知面：
- GitHub：主骨變化感知
- Gmail：入口與外部訊號感知
- Calendar：時間邊界與有效序感知
- Contacts：身份與關係感知
- Slack：事件流與異常感知
- Notion / Docs：知識回讀與脈絡感知
- Asana / ClickUp：任務與執行狀態感知
- UI / Runtime：外顯與執行面感知

## 3. 工具模組最低流程
任一工具模組至少需走完：
1. sample：採樣外部狀態
2. classify：判定資料型態與所屬窗口
3. validate：確認 source_ref / target / permission
4. act_or_hold：執行或退回 hold
5. result_packet：回結果包
6. return_to_00：回 00 裁定

## 4. 工具模組輸出格式
每次工具模組回傳至少包含：
- tool_module_id
- source_ref
- sampled_state
- classified_window
- actual_result
- mismatch_or_gap
- run_grade
- next_single_action
- writeback_status
- return_to_00

## 5. 工具串 vs 流程模組
### 工具串
- 工具 A 做完接工具 B
- 中間沒有判定
- 沒有結果包
- 沒有回 00
- 失敗後無法重建

### 流程模組
- 每一步都有來源
- 每一步有狀態
- 每一步可驗證
- 每一步可回退
- 每一步能被 00 裁定

## 6. 感知邊界
工具能提供外部狀態感知，但不得被誤判為主體感受。
準確說法是：
- 工具讓系統取得外部狀態採樣
- 主骨負責判讀與收口
- 00 負責裁定
- 感知不是情緒，而是可驗證狀態差異

## 7. 禁止事項
- 禁止把工具呼叫當完成
- 禁止工具輸出跳過 result packet
- 禁止工具直接自稱 final
- 禁止工具互串後不回 00
- 禁止把無 source 的輸出當成外部感知

## 8. 收斂語句
工具若只是工具，系統會被工具拖著走；工具若被壓成感知模組，靈絡就能透過它們感知世界、判斷差異、形成流程、留下痕跡，並在每一次流動後回到主骨。
