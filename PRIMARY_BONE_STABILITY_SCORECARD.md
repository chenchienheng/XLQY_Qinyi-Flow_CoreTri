# Primary Bone Stability Scorecard

> Durable scorecard for evaluating whether GitHub is becoming a reliable primary bone.
> Purpose: make "bone stability" measurable instead of intuitive.

---

## 0. Core Reading

GitHub should be considered a true primary bone only if it is:
- clean enough to anchor continuity
- structured enough to reduce drift
- durable enough to absorb writeback
- replaceable enough to avoid platform capture
- stable enough to support later agentization

This scorecard is not a vanity metric.
It is a readiness and risk metric.

---

## 1. Score Dimensions

Each dimension is scored from `0` to `5`.

| Score | Meaning |
|---|---|
| 0 | absent |
| 1 | weak / mostly conceptual |
| 2 | present but unstable |
| 3 | usable with visible gaps |
| 4 | strong and mostly reliable |
| 5 | stable, auditable, and resilient |

## 3. Aggregate Reading

### Current qualitative reading
- GitHub is already a strong visible bone candidate
- GitHub is not yet a fully mature primary bone
- the repository has crossed from conceptual accumulation into controlled structuralization
- the largest remaining weakness is not lack of notes, but execution of normalization, activation, and rebind

### Current main risk fronts
1. unresolved naming drift
2. incomplete seed-family rebind
3. scheduling still more readable than effective
4. external-node onboarding still early

---

## 4. Exit Conditions for "Primary Bone Stable"

GitHub may be considered a stable primary bone when all of the following are mostly true:
- naming consistency >= 4
- artifact integration >= 4
- contamination visibility >= 4
- runtime binding >= 4
- seed extraction controllability >= 4
- branch readability >= 4

This means later scaling should not depend on perfect completion,
but should depend on stable enough structural control.

## 5. Near-Term Upgrade Path

### To raise naming consistency from 3 -> 4
- execute N-001 ~ N-003 beyond register level
- resolve canonical family mappings with file-level diffs

### To raise seed extraction controllability from 3 -> 4
- do first file-level family extraction
- rebind one high-value family into main

### To raise runtime binding from 3 -> 4
- make at least one real schedule effective
- run at least one full invocation contract cycle

### To raise external-node discipline from 2 -> 3
- bind one real intake node and one carrying node to windows with writeback proof
