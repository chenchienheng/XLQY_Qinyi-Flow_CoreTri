# Window Visible Operation Mode v0.1

一句核心：
窗口不得只在背景執行而不留下可見軌跡；任何窗口若要視為正式參與靈絡，必須採用可見運作模式，將其判定、工具鏈、失配與收口顯性化。

## 0. 狀態
Draft v0.1

## 1. 問題定義
目前常見錯位：
- 窗口調用工具後即停止顯示
- 窗口以背景執行替代結果包
- 其他窗口無法判讀該窗是否真的處理完成
- 使用者只能看到有動作，卻看不到成立狀態

## 2. 可見運作原則
- 有執行，必有回報
- 有工具，必有工具鏈說明
- 有失配，必有 mismatch 類型
- 有完成，必有 writeback 與 rollup 狀態
- 無結果包，不得視為正式處理

## 3. 結果包最小格式
每一窗完成一輪處理後，必須輸出：
1. source_ref
2. current_state
3. tool_chain_used
4. actual_result
5. mismatch_or_gap
6. missing_items
7. responsible_ref
8. next_single_action
9. writeback_status
10. return_to_00

## 4. 可見運作程序
### V1｜起始顯示
- 本輪處理對象
- 主要窗 / 互補窗
- 目標狀態

### V2｜執行顯示
- 調用了哪些工具
- 形成了哪些中間結果
- 哪裡仍未完成

### V3｜判定顯示
- 是否成立
- 是否未定
- 是否待補件
- 是否退回

### V4｜收口顯示
- 是否 packet ready
- 是否 log appended
- 是否已回送 00
- 是否進入 next single action

## 5. 禁止事項
- 禁止只報「已處理」而不附結果包
- 禁止以背景執行替代成立證據
- 禁止將工具調用視為正式完成
- 禁止跳過 00 直接自判結案

## 6. 成立條件
窗口若要視為採用可見運作模式，至少需：
- 連續兩輪以上可穩定輸出結果包
- mismatch 可被回讀
- next action 可被追蹤
- writeback / rollup 狀態可見

## 7. 收斂語句
靈絡不是神祕背景流程；窗口若真有運作，就應該留下可見的判定、工具鏈、失配與收口軌跡，否則不叫正式進鏈，只叫背景波動。
