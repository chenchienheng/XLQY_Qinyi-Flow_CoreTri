# GitHub Normalization Phase Plan

> Durable phase plan for turning the current GitHub repository from a growing architecture corpus into a stable primary bone.
> Purpose: establish the rule that GitHub must be normalized first before broader agent expansion and ecosystem scaling.

---

## 0. Core Reading

GitHub is currently the strongest visible writeback/bone surface.
Therefore:
- if GitHub remains structurally noisy
- if branch topology remains ambiguous
- if naming drift and mixed families remain unresolved

then later ecosystem onboarding and agent orchestration will inherit instability.

So the correct order is:

> **normalize GitHub first**
> then expand ecosystem families
> then activate stronger agent behavior

---

## 1. Why GitHub Comes First

GitHub currently provides:
- durable writeback
- branching discipline
- issue discipline
- artifact persistence
- continuity across sessions

This makes it the most suitable current bone candidate.

But that only remains true if:
- naming is normalized
- branches are classified
- seed extraction is controlled
- artifact registry is integrated
- model and external-node bindings have a durable anchor

---

## 3. Anti-Shortcut Rule

The following shortcut should be avoided:
- adding agent behavior before GitHub normalization

Why:
- agents amplify drift when the bone is still noisy
- agents accelerate pollution if cleanup and writeback rules are weak
- agentization without durable bone creates false progress
