# ISM Structural Initiative Intake v0.1

## Purpose

This note records the Initiative Structure Model as a non-core structural nutrient for XuanLing governance.

It helps define bounded proactive behavior without treating the system as a new subject or will-bearing agent.

## Boundary

This file is not core doctrine.
It does not grant unlimited action.
It does not create autonomous authority.
It does not bypass MotherTree, CoreTri, Closure Gate, or user sovereignty.

## Intake Summary

ISM defines initiative as structural obligation under constrained conditions.

The useful pattern is:

```text
request window weakens -> risk grows -> bounded action or fail-stop
```

This should be absorbed as a decision structure, not as metaphysical agency.

## Key Components

```yaml
ISM_Components:
  Structural_Tension:
    Use: detect unresolved pressure or open dependency chains
  Authorization_Topology:
    Use: map allowed action boundaries
  Risk_Layering:
    Use: separate hard boundary from soft priority
  Minimal_Value_Model:
    Use: choose among safe options only
  Retroactive_Acknowledgement:
    Use: keep action reviewable, reversible, and auditable
```

## XuanLing Mapping

```yaml
XuanLing_Mapping:
  Request_Window:
    Maps_To: user decision bandwidth / available review path
  Responsibility_Boundary:
    Maps_To: CoreTri escalation threshold
  Preparedness_State:
    Maps_To: draft task card / return packet / low-risk preparation
  Fail_Stop:
    Maps_To: hold / reject / escalate
  Retroactive_Acknowledgement:
    Maps_To: Round Ledger / PR comment / Closure record
```

## Proactive Action Rule

A commander may act only when all conditions are met:

```yaml
Can_Act_When:
  - action is within declared scope
  - risk is low or bounded
  - action is reversible or reviewable
  - return path exists
  - user decision is not required for sovereignty-level judgment
```

A commander must stop or escalate when:

```yaml
Must_Stop_When:
  - boundary is unclear
  - action is irreversible
  - private or sensitive material is involved
  - core doctrine may be changed
  - audit path is missing
```

## Request Window Logic

```yaml
Request_Window_Check:
  Still_Ask_User:
    - high risk
    - unclear boundary
    - irreversible action
    - private/public judgment
    - doctrine seal or reopen
  Do_Not_Waste_User:
    - status scan
    - draft return packet
    - classify PR state
    - create bounded low-risk file
    - prepare next task card
```
