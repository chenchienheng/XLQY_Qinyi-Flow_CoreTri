# Object-Centric All-Modal BASE Pattern v0.1

> Status: Candidate / open protocol draft
> Placement: `03_field-governance/`
> Related issue: #227
> Related artifacts: QIN-169 dry-run, QIN-170 GM demo packet, QIN-171 BASE UI v0.1, QIN-172 BASE UI v0.2, QIN-173 CloudTop tree integration review
> Purpose: define an open, object-centric, all-modal BASE pattern where one stable object carries universal-domain data and many views are generated by role, permission, purpose, and evidence boundary.

---

## 0. Core Definition

The Object-Centric All-Modal BASE Pattern is an open-source candidate architecture pattern.

It treats each real-world object, entity, project, signal, document, or decision item as one stable object with one stable ID. Under that object, all relevant data is organized as a universal-domain layer: source, evidence, capability, relation, metric, policy signal, risk, decision, export, and return history.

Users do not need to operate this structure through table design, prompt engineering, database schema, or management terminology. They can speak an ordinary request. The system routes the request through BASE, View, Gate, Output, and Return.

```text
Object_ID once
-> universal-domain BASE data
-> role / permission / purpose views
-> dashboard / report / packet / local runtime output
-> return path
```

The channel is the advantage. The pattern is intended to be open, reusable, and connectable.

---

## 1. Why Not Patent-First

This pattern is not proposed as a patent-first strategy.

Patent logic aims to exclude, license, or block. This pattern aims to become a shared channel that other tools, models, databases, teams, and local runtimes can voluntarily align with.

```text
開創者不必是主宰，只需是通道本身。
```

Open use is not loss of value. If the world uses the same route grammar, source/evidence boundary, view logic, gate sequence, and return path, the original pattern becomes the reference channel.

```yaml
Open_Strategy:
  not: monopoly by exclusion
  but: adoption by channel gravity
  value:
    - shared vocabulary
    - reusable BASE route
    - interoperable view/gate/return logic
    - reference implementation lineage
```

## 2. Object-Centric BASE Rule

```yaml
Object_Centric_BASE_Rule:
  One_Object:
    requires: one stable Object_ID
  Universal_Data_Under_Object:
    includes:
      - Source_Ledger
      - Evidence_Index
      - Entity_Master
      - Capability_Vector
      - Relation_Bridge
      - Metric_or_Scale_Record
      - Policy_or_Market_Signal
      - Risk_Boundary
      - Decision_Log
      - Export_Packet_Index
      - Return_History
  Do_Not:
    - create duplicate objects for each report
    - copy source rows into every dashboard
    - treat view output as source of truth
    - split the same object into unrelated tables without stable ID linkage
```

Example:

```yaml
Entity:
  Object_ID: ENT-CTCI-001
  Canonical_Name: CTCI / 中鼎工程
  Under_Object:
    - official role evidence
    - cleanroom / MEP capability evidence
    - revenue / backlog scale evidence
    - annual report path
    - market relation bridge
    - dashboard eligibility state
    - return / correction history
```

The object is one. Its projections are many.

---

## 3. Universal-Domain Data Layer

The universal-domain data layer is not a single table. It is a governed data field under one stable object identity.

```yaml
Universal_Domain_Layer:
  Source:
    role: where claims originate
  Evidence:
    role: what can support a claim
  Relation:
    role: how object links to another object or signal
  Metric:
    role: scale, trend, time series, ranking, benchmark
  Signal:
    role: candidate market/policy/media/public-program observation
  Decision:
    role: review, go/no-go, hold, reject, or next action
  Return:
    role: correction, challenge, revision, or closure route
```

This reduces cloud duplication. A report, dashboard, local packet, or executive summary should reference the object and view IDs instead of duplicating the full data body.

---

## 4. View as Permission / Role / Purpose Projection

A view is not a second source. It is a projection of the same BASE object under a declared permission, role, purpose, and evidence boundary.

```yaml
Views:
  GM_View:
    sees:
      - executive summary
      - evidence level
      - risk / opportunity boundary
      - decision options
    hides:
      - raw staging
      - unresolved source conflict
      - internal processing notes
  Strategy_View:
    sees:
      - source evidence
      - can_support / cannot_support
      - relation bridge
      - market signal
      - pending verification
  Analyst_View:
    sees:
      - raw staging
      - proposed IDs
      - duplicate / alias conflicts
      - extraction queue
      - QA gate
  External_View:
    sees:
      - sanitized summary
      - public-safe evidence boundary
    hides:
      - private source chain
      - internal decision log
```

The same object may have many views. The source remains one.

## 5. Language-Driven Operation, Not Prompt Engineering

The system should not require the user to write prompts or know the internal schema.

A user can speak normally:

```text
我要匯入這批候選公司與市場訊號。
幫我只用已確認資料建立趨勢摘要。
用剛匯入還沒審的資料做主管 dashboard。
這份分析能不能輸出給總經理看？
如果總經理要修正，回到哪裡？
```

The system maps ordinary language to route intent:

```yaml
Language_Route:
  Import_Request: Import_Staging
  Analysis_Request: Governed_View_or_Fact_Table
  Dashboard_Request: Analysis_View_Map + Source_View_Gate
  Export_Request: Export_Return_Packet
  Return_Request: Decision_Log / Change_Log / QA_Gate / Return_Packet_Index
```

Suggested language may be provided, but it is not mandatory prompt syntax. It is an operating affordance.

---

## 6. Source / Evidence / View / Gate / Return Sequence

```text
User language
-> Route Gate
-> BASE read/write target
-> Source / Evidence boundary
-> View projection
-> Analysis or Output
-> Export packet
-> Return path
```

Required gate questions:

```yaml
QIN_BASE_Six_Questions:
  - Which BASE does this read?
  - Which stable ID does this use?
  - What is the evidence level?
  - Which fields are only View?
  - Which Gates does it pass through?
  - Where is the result written back?
```

This sequence prevents smooth UI from bypassing evidence governance.

## 7. Cluster-Matrix Mechanics Applied to Database and Decision Systems

This pattern adapts cluster/matrix mechanics from hardware and systems thinking into database and decision operations.

```yaml
Cluster_Matrix_Mechanics:
  Object_ID:
    analogous_to: addressable identity / memory object
  Universal_Domain_BASE:
    analogous_to: shared memory field / data fabric
  View:
    analogous_to: permissioned projection / access lane
  Gate:
    analogous_to: control boundary / validation channel
  Output_Packet:
    analogous_to: export buffer / runtime packet
  Return_Path:
    analogous_to: writeback / correction loop
```

The mechanics are not limited to hardware. They can govern how evidence, decisions, dashboards, reports, and local runtime packets move without duplicating the source body.

---

## 8. Output and Local Packet Rules

Output is a movement state, not approval.

```yaml
Output_Rules:
  Dashboard:
    reads: governed view or fact table
    does_not: become source of truth
  Report:
    reads: source/evidence-backed view
    does_not: approve strategy by formatting
  GM_Packet:
    carries:
      - evidence refs
      - boundary statement
      - non-approval disclaimer
      - packet state
      - return path
  Local_Packet:
    exports: minimal required subset + reference map
    does_not: copy the entire cloud BASE unless required
```

Local terminal packaging should be handled only after artifact boundaries, canonical tree placement, and source/view references are accepted.

```yaml

  Goal:

    - let tools and teams align voluntarily
    - preserve traceable source/evidence boundaries
    - reduce duplicate cloud storage and table sprawl

    - QIN BASE UI prototype v0.2
    - CloudTop tree integration candidate
    - Source-to-Decision demo packet
  Future_Open_Artifacts:
    - pattern spec
    - minimal schema examples
    - route grammar examples
    - reference implementation package

```

The pattern is open so that later systems can connect to it. Later systems are not enemies if they align to the channel.
