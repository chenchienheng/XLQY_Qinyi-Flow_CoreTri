## 1. One-Line Reading

This is not just a presentation tutorial. It demonstrates a complete Codex work pattern: local habitat, permission gate, outline-first production, route choice, output, and skill recycle.

---

## 2. Core Workflow

```text
Project Folder
→ Permission Gate
→ Source Check
→ Markdown Outline
→ Human Confirm
→ Route Choice
→ Output
→ Manual Review
→ Skill Recycle
```

---

## 3. Local Habitat

Codex work starts with a bounded project folder, not with immediate output generation.

```yaml
Local_Habitat:
  Local: local workspace
  Carrier: project folder
  Boundary: work only inside selected folder
  Authority: ask before modifying important files
  Return: deliver artifacts back into folder
```

Purpose:

- put source files, reports, logos, visual references, and output targets in one bounded area
- make the agent's work surface visible
- prevent unbounded file wandering

---

## 4. Permission Gate

Codex should not be read as a free-running actor.

Permission gate means:

- use bounded project folder
- prefer default or approval-based permissions for first pass
- do not alter files before source check
- require confirmation before production
- do not treat one successful run as permanent permission

---

## 5. Outline First

Core rule:

```text
先定錨，再生成。
先確認資料，再排版。
先大綱，再成品。
```

Minimum source-check prompt logic:

```text
Check the folder first.
Report which files you see and what they likely represent.
Do not modify files yet.
Wait for confirmation before production.
If information is insufficient, ask first instead of generating directly.
```

This prevents beautiful garbage.

---

## 6. Route Choice

Presentation output should follow use case, not visual impulse.

| Route | Use | Strength | Limitation |
|---|---|---|---|
| Editable Route / PPTX | internal review, office work, repeated revision | editable text boxes, easier collaboration | may need manual layout polish |
| Image/PDF Route | external talk, campaign, final visual deck | high visual consistency and impact | text is harder to revise after image generation |

Rule:

```text
A presentation route is a strategy choice, not merely a design choice.
```

---

## 7. Skill Recycle

The most valuable part is not a single good deck.

The value appears when a successful workflow is reviewed and stored as a reusable skill.

```text
successful run
→ review what worked
→ write trigger condition
→ write tools and rules
→ define limits
→ define return format
→ reuse next time
```

This is LOR / Recycle.

If a successful run is not recycled, it remains token cost.
If it becomes a skill, it becomes workflow capital.

---

## 9. Safety Boundaries

- do not modify important original files directly
- do not generate final artifacts before outline confirmation
- do not work outside the selected project folder
- do not treat a single success as a permanent rule
- skill must define trigger, purpose, limits, tools, and return path
- route choice must match audience and revision needs

---

## 10. Relationship to ChatGPT / Qinyi

This module should not be used to claim that Codex is universally stronger than ChatGPT.

Better reading:

```text
Codex is stronger for local file-based artifact production and reusable skill workflows.
ChatGPT / Qinyi is stronger for conversation, reasoning, judgment, translation, and coordination calibration.
```

They are complementary surfaces.
