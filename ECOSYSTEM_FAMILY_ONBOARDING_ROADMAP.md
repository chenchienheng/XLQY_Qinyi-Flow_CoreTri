# Ecosystem Family Onboarding Roadmap

> Roadmap for gradually connecting more ecosystem families into the XLEN / Xuanling runtime without collapsing into single-platform dependence.
> Purpose: make ecosystem expansion staged, auditable, and role-aware.

---

## 0. Core Reading

Tools, clouds, models, and platforms are not the system center.
They are ecosystem families and replaceable nodes.

Therefore onboarding should not mean:
- indiscriminate connector accumulation
- flat multi-tool clutter
- one platform becoming the hidden sovereign center

Instead onboarding should mean:
- family-aware expansion
- role-aware binding
- schedule-aware activation
- writeback-aware continuity

---

## 1. Onboarding Principle

Each ecosystem family should be connected through five questions:

1. what role does it serve?
2. which window owns it?
3. which coupling face does it strengthen?
4. what writeback surface proves its effect?
5. can it be replaced without breaking main continuity?

If these are not answered, onboarding remains incomplete.

## 3. Staged Onboarding Order

### Phase 1 — stabilize current bone and timing
Priority families:
- repository / writeback family
- calendar / time family
- communication / intake family

Reason:
- continuity, cadence, and event intake must stabilize before large ecology expansion

### Phase 2 — connect operational execution families
Priority families:
- workflow / board family
- carrying / knowledge mesh family

Reason:
- once timing and return work, operational routing can safely expand

### Phase 3 — connect presentation and outward reflection
Priority families:
- presentation / external view family

Reason:
- outward readable layers should reflect stable runtime, not replace it

### Phase 4 — expand differentiated model families
Priority families:
- model / reasoning family

Reason:
- multi-model cooperation becomes strongest when the main chain, writeback, timing, and orchestration surfaces already exist

## 6. Anti-Capture Rule

The purpose of onboarding many families is **not** to become dependent on all of them.
The purpose is:
- reduce single-platform capture risk
- preserve replaceability
- let similar tools occupy similar roles without becoming sovereign

Therefore a family is healthy only when:
- it can be useful
- it can be removed
- it can be replaced
- the main chain still survives
