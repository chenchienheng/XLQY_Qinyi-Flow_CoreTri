## 3. Core Learning

### 3.1 Codex is workspace agent, not system body

Codex is not only a coding executor.
It is closer to a project-folder / repo / local-workspace worker.

It may support:

- reading folders
- organizing repos
- producing documents
- forming presentation packs
- building reviewable packs
- artifact review
- QA report generation
- next-round task continuation

It must not be promoted into:

- XuanLing body
- Qinyi replacement
- autonomous runtime
- approved executor
- closeout authority

### 3.2 Contract makes Codex usable

Without contract, Codex can become verbose, high-token, uncontrolled, and branch-heavy.

Codex becomes reviewable only when constrained by:

- LOR
- Gate
- Workstream
- Return Packet
- Artifact Contract
- Stop Condition
- Cost Gate
- Resume Card

Codex's value is not merely doing more. Its value is doing work that can be checked, reverted, continued, and returned.

### 3.3 Local folder is habitat

A local project folder is not just a folder.

It is Codex's Local Habitat:

- source files live there
- artifacts return there
- revisions must be traceable
- OneDrive / local path / repo path carry different risks
- local workspace must not be confused with source of truth
- synced folders may introduce lag, metadata conflict, and accidental propagation

Local Habitat requires health checks.

### 3.4 Artifact verification is required

Artifact output is not deliverable until verified.

Every artifact should ask:

- Review Pack or Per-Source Revised Pack?
- Build-Ready Draft or actual Flow?
- Editable route or visual route?
- manifest present?
- source mapping present?
- diff summary present?
- unresolved risk listed?
- next review need visible?

Artifact without verification is output, not deliverable.

### 3.5 Closeout must include resume logic

Closeout must not merely say complete.

A useful closeout should include:

- Status
- Changed Surface
- Artifact Diff Summary
- External Effects
- Unresolved Risk
- Stop Condition
- Next LOR Ready
- Resume Card

Otherwise it is summary, not closeout.

### 3.6 Token capital is not token minimalism

This round had high token pressure.

It should be read as high token usage with structural recycle because part of the usage returned into:

- Codex custom-instruction candidates
- LOR templates
- Closeout / Resume rules
- Artifact Mode
- Build-Ready criteria
- OneDrive boundary rules
- PC Diagnostic support workflow
- Flow Cell Foundation branch strategy
- Security Scan Context
- Codex Node Calibration Evidence

This was not low-cost, but it was not pure uncontrolled burn.

## 5. New Candidate Valences

### Workspace Agent Valence

Ask which carrier Codex is working in:

- conversation
- repo
- local folder
- synced folder
- cloud workspace

Different carriers have different risks.

### Local Habitat Health Valence

Ask whether the local workspace is healthy:

- PC slowdown
- OneDrive sync state
- Defender scanning repo
- browser RAM pressure
- disk I/O bottleneck

Unhealthy habitat increases cost and instability.

### Artifact Verification Valence

Ask whether output has source, version, manifest, diff, risk, acceptance criteria, and return path.

Without verification, artifact remains output.

### Cost Gate Valence

Ask whether token / tool / runtime cost has budget, stop condition, retry limit, and recycled output.

High token without return is burn.
High token with return may become Token Capital.

### Resume Valence

Ask whether next round can continue after interruption.

A usable resume card preserves state, boundary, unresolved risk, and next step.
