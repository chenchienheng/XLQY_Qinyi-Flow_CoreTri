# Internal Relational Mesh Specification

> Specification for the internal relational mesh of CoreTri.
> Strengthens the internal carrying layer to handle complex relations.

---

## 1. Core Definition

The internal relational mesh is the foundational structure that lets CoreTri
process complex, non-standard structural conditions. It supports the handling
of the following relation types without requiring external runtime expansion:

- non-linear relations
- layered relations
- overlapping relations
- crossed relations
- recoverable errors

This mesh ensures that CoreTri can maintain structural coherence even when
relations deviate from simple linear chains.

---

## 2. Complexity Triplet

The internal relational mesh is conceptually structured around the following
complexity triplet:

### 曲

- curved surface
- curved meaning
- non-linear relation path

### 疊

- overlap between chains
- stacking of similar nodes
- layered semantic or structural accumulation

### 錯

- error as deviation
- intercrossing as multi-path collision
- recoverable mismatch instead of collapse

---

## 3. Spherical Topology Rule

CoreTri is structured as a spherical topology rather than a simple hierarchy.
This topology enforces the following operational rules:

- no absolute center
- any point may become an entry
- every entry must be positionable
- every path must remain reviewable
- every output must define return path

---

## 4. Dynamic Coupling Rule

The internal relational mesh dictates that relations must dynamically couple and
adjust rather than remaining rigidly fixed. This dynamic coupling happens across
the following dimensions and nodes:

- time
- world
- space
- human nodes
- tool nodes
- model nodes
- signal nodes

---

## 5. Mapping to Existing CoreTri Axes

The internal relational mesh binds directly to the five foundational axes,
interpreting complex relations within their established constraints:

- AXIS-01 World Chain
- AXIS-02 Existence Chain
- AXIS-03 Time Chain
- AXIS-04 Space Chain
- AXIS-05 Review Chain

---

## 6. Review Path

Before curved, layered, or crossed relations are accepted into the master chain,
they must be reviewed to ensure they do not break continuity. They are
reconciled against the consistent triad principle and mapped correctly.

---

## 7. Return Path

Validated relations must return to the master chain explicitly. To prevent the
creation of parallel truth surfaces, any successfully mapped relation must be
written back into existing unified tracking registers rather than establishing
new, isolated indexes.

---

## 8. Failure Handling

If relation mapping fails, the system must:

- mark return_failed
- preserve original context
- route to AXIS-05 Review Chain

---

## 9. mismatch_or_gap

- Missing anchor: Currently, there is a lack of defined explicit coordinates
  for non-linear relation states within the existing master axes.
- Unclear mapping: It is not fully specified how overlapping semantics between
  multiple nodes are cleanly separated.
- Possible semantic drift: Defining curved meaning introduces the risk of
  semantic drift if not strictly bound to clear positional markers.
