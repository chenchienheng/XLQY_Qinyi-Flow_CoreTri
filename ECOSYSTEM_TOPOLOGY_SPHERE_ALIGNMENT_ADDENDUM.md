## 1. Core Reading

GitHub is the strongest current bone and writeback anchor, but it is not the whole system.

The broader XuanLing field includes:

- GitHub / repository bone
- ChatGPT / Qinyi conversational surface
- Codex / Jules bounded execution nodes
- Google Drive / knowledge mesh
- M365 / workflow surfaces
- Gmail / communication intake
- Slack / Teams / message surfaces
- ClickUp / Asana / Linear / board families
- Notion / knowledge surfaces
- model families
- image and visual identity tools
- Zenodo / public archival anchor
- web / public release surfaces

These are not flat tools. They are sphere-surface nodes around an invariant core.

---

## 2. XuanLing Cluster Matrix Difference

A normal cluster matrix often groups nodes by technical similarity.

XuanLing cluster reading groups nodes by:

```yaml
XuanLing_Cluster_Read:
  dependency_relation:
  authority_ring:
  coupling_face:
  carrier_shell:
  return_path:
  replacement_safety:
  depth_governance:
  tri_coupled_status:
```

Therefore a tool family is not mature merely because it exists.
It becomes mature only when it can be positioned, bounded, replaced, audited, and returned.

---

## 3. Tri-Coupled Ecosystem Test

Each ecosystem family must pass:

```yaml
Tri_Coupled_Ecosystem_Test:
  Xiangyi_相依:
    ask: what does this family depend on and what depends on it?
  Hucun_互存:
    ask: how does this family cooperate without collapsing into another role?
  Xulian_續鏈:
    ask: where does its output persist, return, and resume?
```

If one is missing, the family remains partial or candidate.

---

## 4. Family Handling Rule

Each family should receive one current handling decision:

```yaml
Family_Handling:
  - keep_active
  - update_boundary
  - bind_to_window
  - bind_to_register
  - hold_candidate
  - archive_trace
  - reject_as_dependency_risk
```

Default rule:

```text
Do not onboard more tools until existing families have role, authority, writeback, and replacement clarity.
```
